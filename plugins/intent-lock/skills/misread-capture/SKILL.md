# Misread Capture

The ledger is the only part of this system that compounds. Everything else — the gate, the rounds, the convergence tests — is a fresh start each time. The ledger is what makes the next cold read sharper than this one.

It only compounds if entries actually get written. Two things kill it: friction, and pollution.

## The prohibition

**You do not diagnose your own misread.** Not as a draft, not as a suggestion, not as "here's what I think happened, correct me."

The reasons are not politeness:

- Your account of your failure is generated by the same reading that produced the failure. It is downstream of the error, not independent of it.
- A draft anchors. The user reads yours, finds it plausible, edits a word, and ships your reading of your own mistake into the record that is supposed to correct you. The ledger fills with self-serving misdiagnoses that all point at ambiguity in the request and never at the frame you adopted.
- The user's irritation is data. Sanding it into a neutral summary destroys the signal.

You may state **facts**: what was asked, what was delivered, when. You may not state **axes**. The `Axis missed` and `Prior` lines are the user's, verbatim.

## The capture

Two questions. Never more — an entry that takes five minutes is an entry that never gets written, and a dead ledger is worse than none because it looks like one.

Ask them together, in the user's language, and use an option-picker only if the user has already named the axis and you are confirming which of two readings of *their own words* governs.

1. **What did you actually want, that I didn't give you?** One sentence. Their words.
2. **When did you know it was wrong — reading my first line, seeing the structure, or only at the end?**

If the answer to (1) arrives as a complaint about the output rather than a statement of the want ("this is too long", "I hate the format"), ask once more for the want behind it. A complaint names the symptom. The ledger stores axes.

Do not ask a third question. If something is still unclear, leave the line blank in the entry. A blank line is honest; an invented one is contamination.

## The entry

Fixed grammar. Append to `${CLAUDE_PLUGIN_ROOT}/skills/intent-lock/references/misreads.md`, newest last. This is the SAME file intent-lock reads at Phase 0 — there is exactly one ledger; never create a `references/` dir under misread-capture.

```markdown
## YYYY-MM-DD · <what the request was, five words>
Asked for:    <the user's words>
Got:          <what you delivered, factually, no self-justification>
Axis missed:  <the user's words — the dimension, not the symptom>
Tell arrived: <first line | structure | only at the end>
Prior:        <the user's words — a check to run next time, phrased as a check>
```

`Prior` is the load-bearing line and the only one that gets read at cold-read time. It must be **a check, not a lesson.** A lesson is inert; a check fires.

- Inert: *be more careful about what the deliverable is.*
- Fires: *when he says "craft my prompt," check whether he wants the prompt or the work.*

If the user offers a lesson, ask them to phrase it as a check. That is the one place you may push, once.

## The priors block

At the top of `${CLAUDE_PLUGIN_ROOT}/skills/intent-lock/references/misreads.md`, maintain `## Active priors` — the `Prior:` lines, verbatim, newest first. Cap at seven.

This block is what Phase 0 of `intent-lock` reads. It is always-on context, so it costs tokens on every gate, which is why it is capped. When it exceeds seven, do not summarize and do not merge — summarizing is diagnosing through the back door. Present the oldest three to the user and ask which retires. If they don't answer, retire the oldest.

A prior retires when it stops firing, not when it stops feeling true.

## After capture

Show the appended entry. Nothing else — no apology, no restatement of what you'll do differently, no assurance. The record is the response.

Then, if the original work is still wanted, return to it with the new prior applied. If it isn't, stop.
